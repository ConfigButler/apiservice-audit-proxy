# FAQ

## Why is there no `install.yaml` in the release?

Many Kubernetes projects ship a pre-rendered `install.yaml` (generated by `helm template`) so users can apply a single file without needing Helm. For this project, that would be actively misleading.

`install.yaml` would be rendered with placeholder default values. The proxy cannot start without several runtime-specific settings:

| Flag | Why it can't be defaulted |
|------|--------------------------|
| `--backend-url` | Points to the real aggregated API server — different for every cluster |
| `--webhook-kubeconfig` | Credential and URL for the audit webhook receiver — unique per deployment |
| `--client-ca-file` | The front-proxy CA from the target cluster — must match the kube-apiserver config |
| TLS serving cert/key | Either cluster-issued (cert-manager) or pre-provisioned — never a safe default |

A user who applied a default `install.yaml` would get a proxy that crashes on startup (`--backend-url is required`) or, worse, one that silently skips identity verification because `--client-ca-file` was not set. Neither outcome is acceptable for a security-sensitive component.

**The Helm chart is the only supported install path.** It forces the caller to supply the required values and validates mutual exclusivity of TLS options at template time. Install it with explicit values:

```bash
helm install apiservice-audit-proxy \
  oci://ghcr.io/configbutler/charts/apiservice-audit-proxy \
  --version <version> \
  --namespace audit-pass-through-system \
  --set proxy.backendURL=https://wardle-server.wardle.svc \
  --set proxy.webhookKubeconfig.secretName=audit-webhook-kubeconfig \
  ...
```

Or supply a values file:

```bash
helm install apiservice-audit-proxy \
  oci://ghcr.io/configbutler/charts/apiservice-audit-proxy \
  --version <version> \
  --namespace audit-pass-through-system \
  -f my-values.yaml
```

## Why does release-please manage versions instead of manual Git tags?

Before release-please, the release job in CI triggered only on manually pushed `v*` tags. That meant:

- The person releasing had to remember to tag locally, push the tag, and hope CI was green.
- No automated changelog — release notes had to be written by hand.
- Version numbers in `Chart.yaml` and `values.yaml` had to be bumped by hand and committed before tagging.

With release-please:

1. Every conventional commit (`feat:`, `fix:`, `docs:`, etc.) merged to `main` is inspected.
2. release-please maintains a "Release X.Y.Z" PR that accumulates all unreleased changes, updates the version in `Chart.yaml`, `values.yaml`, and `CHANGELOG.md`, and keeps it current as more commits arrive.
3. Merging that PR is the only release action required — the tag, GitHub release, Docker image push, and Helm OCI push all happen automatically.

The version bump follows [Conventional Commits](https://www.conventionalcommits.org/):

| Commit prefix | Version bump |
|---------------|-------------|
| `fix:` | patch (0.1.0 → 0.1.1) |
| `feat:` | minor (0.1.0 → 0.2.0) |
| `feat!:` or `BREAKING CHANGE:` | major (0.1.0 → 1.0.0) |
| `chore:`, `docs:`, `ci:`, etc. | no bump (hidden in changelog) |
